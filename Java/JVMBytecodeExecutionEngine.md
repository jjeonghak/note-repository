# 바이트코드 실행 엔진
가상 머신의 실행 엔진은 순수하게 소프트웨어로만 구현, 따라서 명령어 집합의 구조와 실행 엔진은 물리적 제약 없음  
실행 엔진의 동작 방법은 `해설 실행`과 `컴파일 실행` 중 하나  

<br>

## 런타임 스택 프레임 구조
자바 가상머신은 메서드를 가장 기본적인 실행 단위로 사용  
메서드 호출과 실행을 뒷바침하는 내부 데이터 구조로 스택 프레임을 이용  
스택 프레임 각각에는 지역 변수 테이블, 피연산자 스택, 동적 링크, 메서드 반환 주소 등 정보 보유  
스택 프레임에 할당해야 하는 메모리 크기는 프로그램 실행 중에는 영향을 받지 않음  
오로지 프로그램 소스 코드와 특정 가상 머신 구현의 스택 메로리 레이아웃에 달려 있음  

<br>

<img width="500" height="300" alt="stack_frame_model" src="https://github.com/user-attachments/assets/9097e12b-482b-47e4-a547-ab6cd3a39f91" />

자바 프로그램 관점에서는 특정 시점에 호출 스택에 쌓여 있는 메서드는 모두 현재 실행 중인 상태  
하지만 실행 엔진 관점에서는 활성 스레드의 스택 맨 위에 있는 메서드만 실행 중인 상태  

<br>

### 지역 변수 테이블
메서드 매개 변수와 메서드 안에서 정의된 지역 변수를 저장하는 공간  
최대 용량은 메서드의 Code 속성 중 `max_locals` 항목에 기록  
메서드 호출 시 매개 변수들도 지역 변수 테이블을 통해 전달  

<br>

<img width="500" height="150" alt="parameter_order" src="https://github.com/user-attachments/assets/32c451ed-027b-42c9-90f1-6cfa032d16c4" />

기본적으로 0번째 변수 슬롯에는 메서드가 속한 객체 인스턴스 참조를 전달  
나머지 매개 변수들부터 채운 후 메서드 본문에서 정의한 변수들이 정의 순서와 유효 범위에 따라 할당  
변수 슬롯을 재사용하면 스택 프레임의 공간을 절약할 수 있지만 부작용 존재  

<br>

```java
// 가비지 컬렉션 시점에 유효 범위이기 때문에 메모리 해제 안됨
public static void main(String[] args) {
  byte[] placeholder = new byte[64 * 1024 * 1024];
  System.gc();
}

// 메모리 해제 안됨
public static void main(String[] args) {
  // 중괄호를 이용한 범위 제한
  {
    byte[] placeholder = new byte[64 * 1024 * 1024];
  }
  // 유효 범위를 벗어났지만 그 후 지역 변수 테이블에 따로 기록한 게 없으므로 변수 슬롯이 그대로 유지
  System.gc();
}

// 올바르게 메모리 해제
public static void main(String[] args) {
  {
    byte[] placeholder = new byte[64 * 1024 * 1024];
  }
  int a = 0;
  System.gc();
}
```

지역 변수 테이블의 변수 슬롯이 여전히 `placeholder`가 가리키는 배역 객체로의 참조를 담고 있는가에 따라 상이  
일부 시간을 많이 소모하는 동작에 앞서 메모리를 대량 점유하는 변수에 수동으로 `null`을 할당하는 코드가 의도적인 코드  
하지만 이보다 변수 범위를 적절히 지정하여 변수가 회수되는 시간을 제어하는게 가장 이상적  

<br>

### 피연산자 스택
메서드가 막 실행된 시점에는 해당 메서드의 피연산자 스택이 비어 있음  
피연산자 스택에 있는 원소의 데이터 타입은 바이트코드 명령어 순서와 정확히 일치 필수  
개념 모델에서 서로 다른 메서드의 가상머신 스택에 있는 스택 프레임들은 완전히 독립적  
하지만 대부분의 가상머신에서는 최적화 과정에서 스택 프레임들을 부분적으로 겹쳐 사용
