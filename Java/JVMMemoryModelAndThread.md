# 자바 메모리 모델과 스레드

## 하드웨어에서의 효율과 일관성
단순히 프로세서의 컴퓨팅(연산)만으로 이루어지지 않고 적어도 메모리는 반드시 필요  
메모리 I/O 작업을 없애기는 매우 어렵고 중간에 캐시 계층을 하나 이상 설정  
하지만 캐시 일관성(`cache coherence`)이라는 새로운 문제 발생  

<br>

<img width="500" height="200" alt="cache_and_processor" src="https://github.com/user-attachments/assets/7ca8e19b-b7d2-48d7-b2e2-4e0f28f689da" />

일관성 문제를 해결하려면 프로세서가 캐시를 이용할 때 정해진 프로토콜 준수 필수  
메모리 모델은 특정 프로토콜을 이용하여 특정 메모리나 캐시를 읽고 쓰는 절차  
또한 비순차 실행 최적화로도 프로세서의 컴퓨팅 능력 향상 가능

## 자바 메모리 모델
하드웨어와 운영 체제의 메모리 모델을 직접 사용하는 C/C++ 언어와 다르게 자바는 자체 메모리 모델을 정의  
여러 스레드가 메모리에 동시에 접근해도 모든 작업이 명확하게 이루어지도록 엄격하게 정의  

<br>

### 메인 메모리와 작업 메모리
자바 메모리 모델의 주된 목적은 프로그램에서 다양한 변수에 접근하는 규칙을 정하는 것  
자바 메모리 모델은 모든 변수가 메인 메모리에 저장된다고 규정  
각 스레드는 자체 작업 메모리를 보유, 작업 메모리는 프로세서의 캐시와 비슷한 역할  

<br>

<img width="500" height="200" alt="thread_work_memory" src="https://github.com/user-attachments/assets/e9068bda-ecf8-46ee-8902-847c11837e3a" />

메인 메모리와 작업 메모리 개념은 자바 힙, 스택, 메서드 영역과 아무런 관련 없는 구분 방식  
굳이 비교하자면 메인 메모리는 자바 힙 중 객체 인스턴스 데이터 부분, 작업 메모리는 가상 머신 스택 영역의 일부  
더 기본적인 수준에서는 메인 메모리는 하드웨어 메모리, 작업 메모리는 레지스터와 캐시에 대응  

<br>

### 메모리 간 상호 작용
메인 메모리에서 작업 메모리로 변수를 복사하고 작업 메모리 내용을 메인 메모리로 다시 동기화하는 과정  
- 잠금(`lock`): 메인 메모리에 존재하는 변수를 특정 스레드만 사용 가능한 상태로 변경
- 잠금 해제(`unlock`): 잠겨 있는 변수를 잠금 해제
- 읽기(`read`): 뒤이어 수행되는 적재 연산을 위해 메인 메모리 변수값을 특정 스레드의 작업 메모리로 전송
- 적재(`load`): 읽기 연산으로 메인 메모리에서 얻어온 값을 작업 메모리의 변수에 복사
- 사용(`use`): 작업 메모리의 변수값을 실행 엔진으로 전달
- 할당(`assign`): 실행 엔진에서 받은 값을 작업 메모리의 변수에 할당
- 저장(`store`): 뒤이어 수행되는 쓰기 연산을 위해 작업 메모리 변수값을 메인 메모리로 전송
- 쓰기(`write`): 저장 연산으로 작업 메모리에서 얻어온 값을 메인 메모리 변수에 기록

<br>

메인 메모리에서 작업 메모리로 변수를 복사하려면 읽기와 적재 순으로 수행  
반대로 작업 메모리의 변수를 다시 메인 메모리로 동기화하려면 저장과 쓰기 순으로 수행  
하지만 꼭 바로 이어서 수행될 필요없음  
- 읽기와 적재, 저장과 쓰기는 단독으로 수행 불가
- 스레드는 최근 할당 연산 버리기 불가, 즉 작업 메모리에서 변수값이 변경되면 메인 메모리로 동기화 필수
- 스레드는 작업 메모리의 데이터를 아무 이유 없이 메인 메모리로 동기화 불가
- 변수는 메인 메모리에서만 새로 발생 가능
- 작업 메모리에 있는 초기화되지 않은 변수를 바로 사용 불가, 즉 변수를 사용 또는 저장 전에 할당과 적재 필수
- 변수를 잠그면 작업 메모리의 변수값은 지워짐, 실행 엔진이 변수를 사용하려면 적재 또는 할당을 다시해서 변수값 초기화 필수
- 잠근 변수에 대해서 다른 스레드가 해제 연산 불가
- 잠금을 해제하려면 변수를 메인 메로리로 동기화 필수(저장과 쓰기)

<br>

### volatile 변수용 특별 규칙
자바 가상 머신이 제공하는 가장 가벼운 동기화 메커니즘  

<br>

모든 스레드에서 이 변수를 투명하게 조회 가능  
가시성을 보장, 즉 한 스레드가 값을 수정하면 다른 스레드들도 새로운 값을 즉시 조회 가능  
하지만 자바의 산술 연산자가 원자적이 아니라 완벽하게 안전하지 못함


```java
public class VolatileTest {
  public static volatile int race = 0;

  public static void increase() {
    // 명령어 한 개가 반드시 원자적으로 수행된다는 보장 없음
    race++;
  }

  public static final int THREADS_COUNT = 20;

  public static void main(String[] args) {
    Thread[] threads = new Thread[THREADS_COUNT];
    for (int i = 0; i < THREADS_COUNT; i++) {
      threads[i] = new Thread(new Runnable() {
        @Override
        public void run() {
          for (int i = 0; i < 10000; i++) {
            increase();
          }
        }
      });
      threads[i].start();
    }

    // 다른 모든 스레드가 종료할 때까지 대기
    while (Thread.activceCount() > 1)
      Thread.yield();

    // 결과는 200,000보다 작은 값 출력
    System.out.println(race);
  }
}
```

<br>

해당 시나리오는 락을 활용하여 원자성 보장 필수  
- 연산 결과가 변수의 현재 값과는 무관하거나 변수의 값을 수정하는 스레드가 하나뿐임을 보장
- 다른 상태 변수와 관련한 불변성 제약 조건에 관여하지 않음

<br>

아래와 같은 시나리오에 매우 적합  

```java
volatile boolean shutdownRequested;

public void shutdown() {
  shutdownRequested = true;
}

public void doWork() {
  // shutdown 메서드가 실행되면 바로 종료됨을 보장
  while(!shutdownRequested) {
    // 비즈니스 로직
  }
}
```

<br>

명령어 재정렬 최적화를 방지  
일반 변수는 메서드 실행 중 할당 결과를 이용해야 하는 모든 위치에서 올바른 결과를 얻는다는 점만 보장  
변수 할당 작업의 실행 순서가 프로그램 코드 순서와 같다는 보장 없음  

```java
Map configOPtions;
char[] configText;

volatile boolean initialized = false;

// 스레도 A 실행
configOptions = new HashMap();
configText = readConfigFile(fileName);
processConfigOptions(configText, configOPtions);
initialized = true;

// 스레드 B 실행
while (!initialized) { // 스레드 A가 설정 초기화를 마칠 때까지 대기
  sleep();
}
doSomethingWithConfig();
```

### long과 double 변수용 특별 규칙
자바 메모리 모델에서는 잠금, 잠금 해제, 읽기, 적재, 할당, 사용, 저장, 쓰기 연산이 모두 원자적 필수  
하지만 64비트 데이터 타입인 `long`과 `double`은 좀 더 느슨한 특별 규칙 적용  
volatile로 지정되지 않은 64비트 데이터의 읽기와 쓰기는 32비트 연산 2개로 나누어 처리 가능  
가상머신이 원자성을 보장할지를 선택하는 이러한 현상을 `비원자적 처리`라고 표현  
JDK 9부터 핫스팟 가상머신은 모든 데이터 타입에 원자적으로 접근하도록 하는 실험적 매개 변수 `-XX:+AlwaysAtomicAccesses` 추가  

<br>

### 원자성, 가시성, 실행 순서
자바 메모리 모델이 직접 보장하는 원자적 변수 연산은 읽기, 적재, 할당, 사용, 저장, 쓰기  
`long`과 `double`은 비원자적일 수 있지만 드문 예외  
애플리케이션 수준의 원자성을 보장해야 할 경우 자바 메모리 모델은 잠금과 잠금 해제 연산을 제공  
`synchronized` 키워드로 동기화한 블록은 `monitorenter`와 `monitorexit` 바이트코드 명령어 사용  

<br>

자바 메모리 모델은 변수값이 수정되면 새로운 값을 메인 메모리에 즉시 동기화  
작업 메모리의 변수값을 읽을 때 메인 메모리로부터 값을 갱신하는 식으로 가시성 확보  
일반 변수든 `volatile` 변수든 마찬가지이지만 멀티스레드 환경의 가시성은 `volatile` 변수가 보장  
`synchronized` 블록의 가시성은 변수의 잠금을 해제하기 전에 변수의 값을 메인 메모리로 다시 동기화라는 규칙으로 확보  
`final` 필드는 생성자에서 초기화, 생성이 완벽하게 끝나지 않은 객체의 참조를 다른 스레드에 전달 불가  

<br>

현재 스레드에서 보면 모든 연산이 순서대로 수행, 하지만 다른 스레드에서 보면 순서가 다를 가능성 존재  
`명령어 재정렬`과 작업 메모리와 메인 메모리 사이의 `동기화 지연` 현상을 의미  
`volatile`은 그 자체에 명령어 재정렬 금지라는 의미 포함  
`synchonized`는 락을 소유한 단 하나의 스레드만 변수 접근 가능 규칙에 의거해 작업 순서 선정  

<br>

### 선 발생 원칙
데이터 경합 발생 여부와 스레드 안전성을 확인하는데 매우 유용한 수단  
선 발생이란 자바 메모리 모델에서 정의된 두 작업의 수행 순서 관계를 의미  
자바 언어는 어떠한 동기화 수단 지원 없이도 아래의 선 발생 규칙들을 보장  
- 프로그램 순서 규칙: 한 스레드 안에서는 제어 흐름 순서에 따라 앞의 연산이 뒤따르는 연산보다 선 발생  
- 모니터 락 규칙: 잠금 해제 연산은 같은 락에 대한 잠금 연산보다 선 발생  
- 휘발성 변수 규칙: `volatile` 변수의 쓰기 연산은 같은 변수에 대한 읽기 연산보다 선 발생
- 스레드 시작 규칙: Thread 객체의 start() 메서드는 해당 스레드의 어떤 작업보다도 선 발생
- 스레드 종료 규칙: 스레드의 모든 작업은 해당 스레드의 종료 감지보다 선 발생
- 스레드 인터럽트 규칙: 인터럽트되는 스레드가 인터럽트 이벤트 발생 감지보다 선 발생
- 종료자 규칙: 객체 초기화는 `finalize()` 메서드 시작보다 서 반생
- 전이성: 연산 A가 연산 B보다 선 발생하고, 연산 B가 연산 C보다 선 발생한다면, A가 C보다 선 발생

<br>

```java
private int value = 0;

// 스레드 A가 호출
public void setValue(int value) {
  this.value = value;
}

// 스레드 B가 호출
public int getValue() {
  return this.value;
}
```

두 메서드는 서로 다른 스레드에 의해 호출되므로 프로그램 순서 규칙은 적용되지 않음  
동기화 블록이 없기 때문에 잠금 및 잠금 해제 연산이 수행되지 않으므로 모니터 락 규칙도 적용되지 않음  
value 변수는 `volatile` 변수가 아니므로 휘발성 변수 규칙 적용되지 않음  
이어지는 스레드 시작, 종료, 인터럽트 규칙과 종료자 규칙 역시 적용되지 않음  
적용 가능한 선 발생 규칙이 없기 때문에 마지막 전이성도 적용되지 않음  
결국 스레드 A가 B보다 먼저 수행되더라도 B가 getValue()로 얻는 결과는 알 수 없음  

<br>

이 작업을 스레드 안전하게 만드려면 모니터 락 규칙을 적용하도록 게터/세터에 `synchronized` 적용 또는 value에 `volatile` 적용  
시간 상 먼저 발생하는 작업이 선 발생하는 작업을 의미하지 않는다는 결론 가능  
반대로 선 발생하면 시간상 먼저 발생해야 한다는 추론도 명령어 재정렬로 인해 불가능  
기본적으로 시간 순서와 선 발생 원칙 사이에는 인과 관계가 없음  

<br>

## 자바와 스레드
동시성이라고 해서 반드시 멀티스레딩을 뜻하진 않음  
하지만 자바에서 동시성은 기본적으로 스레드와 분리할 수 없음  

<br>

### 스레드 구현
스레드는 프로세스보다 가벼운 스케줄링 단위  
스레드 각각은 프로세스 자원을 공유 가능하고 독립적으로 스케줄링  
`start()` 메서드가 호출된 후 아직 종료되기 전인 모든 `java.lang.Thread` 클래스의 인스턴스가 바로 하나의 스레드  
대부분의 다른 자바 클래스 라이브러이아 다르게 핵심 메서드 모두가 네이티브 코드로 구현  
스레드 구현 방식은 커널 스레드 구현(1:1), 사용자 스레드 구현(1:N), 사용자 스레드와 경량 프로세스의 하이브리드 구현(M:N)  

<br>

### 커널 스레드 구현
커널 스레드를 이용한 구현은 `1:1 구현`  
커널 스레드는 운영 체제 커널에서 직접 지원하는 스레드, 스레드의 작업을 각 프로세서에 매핑하는 역할  
프로그램은 일반적으로 커널 스레드를 직접 사용하지 않고 커널 스레드의 고수준 인터페이스인 경량 프로세스를 사용  
경량 프로세스가 우리가 일반적으로 부르는 스레드

커널 스레드를 기반으로 구현되기 때문에 생성, 소멸, 동기화 등 다양한 스레드 연산이 시스템 호출로 이루어짐  
경량 프로세스 하나가 커널 스레드 하나에 매핑되기 때문에 경량 프로세스는 일정량의 커널 자원을 소모(시스템에 따라 경량 프로세스 개수 제한)  

<br>

### 사용자 스레드 구현
사용자 스레드를 이용한 구현은 1:N 구현  
넓은 의미로 커널 스레드가 아닌 모든 스레드는 일종의 사용자 스레드  
좁은 의미로 온전히 사용자 공간에서 구현되는 스레드 라이브러리를 가리킴  
운영 체제 커널은 사용자 스레드의 존재와 구현 방법을 알지 못함  
사용자 스레드의 생성, 소멸, 동기화, 스케줄링은 모두 커널의 도움없이 온전히 사용자 공간에서 처리  

<br>

<img width="500" height="250" alt="user_thread" src="https://github.com/user-attachments/assets/f0f4c452-aacb-45fe-9c04-3424b64fd2c9" />

장점이자 단점은 시스템 커널의 지원을 받지 못한다는 것  
운영체제는 프로세서 자원을 프로세스에만 할당하기 때문에 `블로킹 처리`와 멀티프로세서 시스템에서 스레드를 `특정 프로세서에 매핑` 문제 해결 어려움
사용자 스레드로 구현된 프로그램은 일반적으로 복잡해서 일반적인 애플리케이션에서는 잘 사용안함  

<br>

### 하이브리드 구현
커널 스레드와 사용자 스레드를 함께 사용하는 방법, 사용자 스레드와 경량 프로세스가 공존  
사용자 스레드의 시스템 호출은 온전히 경량 프로세스에 의해 수행되므로 프로세스 전체가 완전히 블록될 위험 줄임

### 자바 스레드 스케줄링
스레드 스케줄링은 시스템이 프로세서 사용 권한을 스레드에 할당하는 작업  
협력적 스케줄링과 선점형 스케줄링 존재  

<br>

협력적 스케줄링 시스템에서는 스레드 실행 시간을 스레드가 스스로 제어  
일을 마친 스레드는 다른 스레드로 전환되도록 시스템에 적극적으로 알림  
구현하기 쉽고 동기화 문제가 발생하지 않음  
하지만 스레드 실행 시간을 제어할 수 없음  

<br>

선점형 스케줄링 방식에서는 각 스레드의 실행 시간을 시스템이 할당  
각 스레드들은 전환 시점을 스스로 결정하지 못함  
자바는 선점형 스케줄링 방식을 이용  
시스템이 자바 스레드의 스케줄링을 자동으로 수행하지만 우선순위를 통해 특정 스레드에 더 많거나 적게 실행 시간 권고 가능  
자바 언어에서는 `Thread.MIN_PRIORITY`부터 `Thread.MAX_PRIORITY`까지 총 10단계로 정의  

<br>

<img width="500" height="200" alt="java_thread_priority" src="https://github.com/user-attachments/assets/30b8e8df-29ea-41c5-aa64-108f04a9346a" />

하지만 결국 결정하는 주체는 운영 체제이기 때문에 그리 믿을 만한 조율 방법은 아님  
운영 체제의 우선순위 단계가 자바 스레드의 단계보다 적다면 문제 발생  

<br>

### 상태 전이
자바 언어에서 스레드 상태는 총 여섯 가지  

<img width="500" height="300" alt="java_thread_state" src="https://github.com/user-attachments/assets/035189a2-e5bc-4579-bbb3-8e0a144bf6a1" />

- 신규: 스레드 생성 후 시작하기 전 상태
- 실행중: 운영 체제 스레드의 상태 중 실행중과 준비에 해당
- 무기한 대기: 다른 스레드가 명시적으로 깨워 주기를 기다리는 상태
- 시간 제한 대기: 일정 시간이 지나면 시스템에 의해 자동으로 깨어나는 상태
- 블록: 배타적 락을 얻기를 기다리는 상태  
- 종료: 스레드 실행 종료

<br>

## 자바와 가상 스레드
다양한 운영 체제별 스레드 모델의 차이를 숨기는 통합된 스레드 인터페이스 제공  

<br>

### 커널 스레드의 한계
자바의 기존 동시성 프로그래밍 메커니즘은 마이크로서비스 아키텍처와 어울리지 않음  
과거 모놀리식 애플리케이션에서는 요청 하나의 시간이 상대적으로 길었기 때문에 스레드 전환 비용이 크게 부각되지 않음  
하지만 이제는 요청당 실행 시간이 매우 짧아지고 수가 많아졌기 때문에 심각한 낭비 초래  

<br>

### 가상 스레드: 자바 해법

<img width="500" height="200" alt="virtual_thread" src="https://github.com/user-attachments/assets/e4a41246-d55f-4209-b4f6-7701087b84e3" />

가상 스레드는 플랫폼 스레드와 N:1 관계, 가상 스레드 하나가 블록되면 플랫폼 스레드는 연결된 다른 가상 스레드의 작업을 이어서 진행  
해당 방식으로 커널 스레드는 문맥 전환 없이 쉬지 않고 애플리케이션 코드를 실행
