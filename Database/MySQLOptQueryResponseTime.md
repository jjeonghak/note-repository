# 쿼리 응답 시간
MySQL의 성능은 곧 쿼리 응답 시간  

<br>

## 쿼리 보고
- 소스: 쿼리 메트릭은 2개의 소스에서 비롯
- 집계: 쿼리 메트릭값은 정규화된 SQL 문법을 기준으로 그룹화되고 집계
- 보고: 쿼리 보고서는 고급 프로파일과 쿼리에 특화된 보고서로 구성

<br>

### 소스
쿼리 메트릭은 슬로 쿼리 로그나 성능 스키마에서 비롯  
슬로 쿼리 로그는 디스크에 있는 로그 파일  
슬로 쿼리 로그는 이전에 1초를 최솟값으로, n초 이상 실행되는 쿼리들만 기록  
현재는 최솟값이 0이며 모든 쿼리를 기록 가능  
성능 스키마는 `performance_schema`와 같은 데이터베이스  

<br>

### 집계
쿼리 메트릭은 쿼리별로 그룹화되고 집계  
쿼리를 어떻게 고유하게 식별하여 쿼리가 속한 그룹을 결정하는지 중요  
이를 해결하는 방법은 정규화된 SQL 문을 `SHA-256` 해시로 변환  

```sql
SELECT col FROM tbl WHERE id = 1;                                 # SQL 문
SELECT `col` FROM `tbl` WHERE `id` = ?                            # 다이제스트 텍스트(정규화된 SQL문)
f49d50dfab1c364e622d1e1ff54bb12df436be5d44c464a4e25a1ebb80fc2f13  # 다이제스트 해시
```

<br>

다른 구문으로 같은 논리의 쿼리를 동적으로 생성하면 서로 다른 다이제스트로 정규화되어 다른 쿼리로 보고  

```sql
# 논리적으로 같아도 서로 다른 다이제스트로 정규화
SELECT name FROM captains WHERE last_name = 'Picard';
SELECT name FROM captains WHERE last_name = 'Picard' AND first_name = 'Jean-Luc';

# 서로 같은 다이제스트로 정규화, SELECT `name` FROM `star_ships` WHERE `class` IN (...)
SELECT `name` FROM star_ships WHERE class IN ('galaxy');
SELECT `name` FROM star_ships WHERE class IN ('galaxy', 'intrepid');
```

<br>

### 보고
각 쿼리에는 많은 메트릭이 존재하며 최소, 최대, 평균, 백분위수 등 여러 통계값이 존재  
또한 샘플, EXPLAIN 계획, 테이블 구조 등의 메타데이터 존재  
쿼리 메트릭 도구는 쿼리 프로파일(`query profile`)과 쿼리 보고서(`query report`) 2가지 수준의 계층 구조로 데이터를 제공  

<br>

쿼리 프로파일에 느린 쿼리가 표시  
쿼리 다이제스트와 함께 메트릭의 제한된 하위 세트를 제공하므로 프로파일이라고 부름  
- 쿼리 총시간: 실행 시간의 총합
- 실행시간 비율: 쿼리 총시간을 실행 총시간으로 나눈 값 
- 쿼리 부하: 쿼리 총시간을 클럭 타임으로 나눈 값

<br>

쿼리 보고서는 하나의 쿼리에 대해 알아야 할 모든 것을 보여줌  
소스에서 비롯된 모든 쿼리 메트릭과 최소, 최대, 평균, 백분위수 등 기본 통계값 포함  
전체 보고서에는 쿼리 샘플, EXPLAIN 계획, 테이블 구조 등의 메타데이터 포함  

<br>

## 쿼리 분석
목표는 느린 응답 시간 해결이 아닌 쿼리 실행에 대한 이해  
가장 중요한 것은 숫자가 아닌 메트릭과 메타데이터에 따른 전체 상황 이해  

<br>

### 쿼리 시간
쿼리 시간에는 또 다른 메트릭인 잠금 시간이 포함  
잠금 시간은 쿼리 시간에 내재된 시간이므로 쿼리 시간이 잠금 시간을 포함한다는 것은 당연  
모든 쿼리는 트랜잭션 안에서 수행되므로 트랜잭션은 최상위 이벤트

### 잠금 시간
쿼리를 실행하는 동안 잠금을 획득하여 사용한 시간  
성능 스키마의 잠금 시간에는 InnoDB의 로우 락 대기가 포함되지 않음  
MySQL 서버의 테이블과 메타데이터 락 대기만을 포함  
슬로 쿼리 로그의 잠금 시간에는 메타데이터, 테이블, 행과 같은 모든 락 대기가 포함  
`SELECT ... FOR UPDATE`, `SELECT ... FOR SHARE`가 아닌 모든 읽기는 비잠금 읽기  
비잠금 읽기는 로우 락을 획득하지 않더라도 메타데이터 락과 테이블 락을 흭득하므로 잠금 시간이 0은 아님  
공유 메타데이터 락은 다른 공유 메타데이터 락과 호환되지만 배타 메타데이터 락은 다른 모든 메타데이터 락을 차단  
`ALTER TABLE`은 배타 메타데이터 락을 획득하는 일반적인 작업  

<br>

### 조회된 행
쿼리 조건 절에 일치하는 행을 찾으려고 접근한 행의 수  
쿼리와 인덱스 선택도를 나타내며 이 값이 높을수록 일치하지 않는 행을 조회하는 데 낭비한 시간이 줄어듬  

<br>

### 보낸 행
클라이언트에 반환된 행의 수  
조회된 행과 비교를 통해 쿼리의 성능을 유추 가능  
- 보낸 행 = 조회된 행: 가장 이상적인 경우는 전체 행의 백분율로 계산했을 때 상대적으로 값이 작은 경우
- 보낸 행 < 조회된 행: 쿼리나 인덱스의 선택도가 좋지 않는 경우
- 보낸 행 > 조회된 행: 드문 경우, 쿼리를 MySQL이 최적화 가능할 때만 가능

<br>

### 영향받은 행
삽입, 갱신, 삭제된 행의 수  

<br>

### 셀렉트 스캔
첫 번째로 접근한 테이블에서 수행한 풀 테이블 스캔 횟수  
쿼리가 인덱스를 사용하지 않는다는 것을 의미하며 일반적으로 성능에 좋지 않음  

<br>

### 셀렉트 풀 조인
조인된 테이블을 대상으로 풀 테이블 스캔 횟수  
셀렉트 스캔과 유사하지만 더 안좋은 상황  

<br>

### 디스크에 생성된 임시 테이블
쿼리가 메모리에 임시 테이블을 만드는 것은 정상이지만 너무 커지면 디스크에 생성  
디스크에 임시 테이블을 만드는 것은 MySQL이 기피  

<br>

### 쿼리 카운트
쿼리 실행 횟수, 이 값이 매우 낮고 쿼리가 느리지 않은 경우 기준이 없고 임의적  

<br>

## 쿼리 응답 시간 개선

<br>

### 직접 쿼리 최적화
쿼리와 인덱스를 변경하는 작업  
이러한 변경 작업은 매우 효과적이라 종종 직접 쿼리 최적화 단계에서 종료될 가능성 높음  
- 인덱스 머지 최적화
- 해쉬 조인 최적화
- 인덱스 컨디션 푸시다운 최적화
- 다중 범위 읽기 최적화
- Constant-Folding 최적화
- IS NULL 최적화
- ORDER BY 최적화
- GROUP BY 최적화
- DISTINCT 최적화
- LIMIT 쿼리 최적화

<br>

### 간접 쿼리 최적화
데이터와 접근 패턴을 변경하는 작업  
쿼리 변경 대신 쿼리가 접근하는 대상과 방법을 변경
