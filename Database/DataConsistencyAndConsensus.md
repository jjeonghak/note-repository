# 일관성과 합의

### 일관성 보장
복제 데이터베이스는 대부분 최소한 최종적 일관성을 제공  
데이터베이스 별 불일치는 일시적이며 결국 스스로 해소  
최종적 일관성은 매우 약한 보장  

<br>

## 선형성
선형성은 최신성 보장(`recency guarantee`), 최종적 일관성보다 높은 보장  
선형성의 기본 아이디어는 시스템에 데이터 복사본이 하나만 존재하고 그 데이터를 대상으로 수행하는 모든 연산은 원자적  
선형성 시스템에서는 클라이언트가 쓰기를 성공적으로 완료하면 그 데이터베이스를 읽는 모든 클라이언트는 그 값을 조회 가능  

<br>

비선형성 시스템은 실시간 데이터 갱신을 보장하지 않기 때문에 혼란을 야기  

<br>

선형성 시스템에서 쓰기 연산이 진행 중에 어떤 클라이언트가 새로운 값을 받았다면 다른 클라이언트도 새로운 값을 받아야함  

<br>

### 잠금과 리더 선출
단일 리더의 리더를 선출하는 과정은 잠금을 사용하는 것이며 이때 잠금은 선형적 필수  
분산 잠금과 리더 선출을 구현하기 위해 아파치 주키퍼와 etcd 같은 코디네이션 서비스 사용  

<br>

### 제약 조건과 유일성 보장
유일성 제약 조건은 데이터베이스에 흔하며 이 제약 조건을 강제하려면 선형성 필요  
잠금과 유사하며, 연산은 원자적 `compare-and-set`과도 매우 비슷  

<br>

### 선형성 시스템 구현
근본적으로 데이터 복사본이 하나만 있는 것처럼 동작하고 그 데이터에 실행되는 모든 연산은 원자적  
- 단일 리더 복제(선형적이 될 가능성 존재): 동기식으로 갱신된 팔로워는 선형적이 될 가능성 존재
- 합의 알고리즘(선형적): 단일 리더 복제와 유사, 스플릿 브레인과 복제본의 늦은 갱신을 막을 수단 포함
- 다중 리더 복제(비선형적): 리더들은 동시 쓰기를 처리하고 비동기로 다른 노드에 복제하기 때문에 비선형적
- 리더 없는 복제(비선형적): 정족수 읽기와 쓰기를 요구하나 최종 쓰기 승리 충돌 해소 방법이 비선형적

<br>

### CAP 정리
애플리케이션이 선형성을 요구하고 네트워크 문제로 일부 복제 서버 연결이 끊기면 회복될 때까지 오류 반환(가용성 없음)  
따라서 선형성이 필요 없는 애플리케이션은 네트워크 문제에 더 강인함  

<br>

때때로 일관성(`Consistency`), 가용성(`Availability`), 분단 내성(`Partition tolerance`) 중 두 개만 선택 강요  
네트워크 분단은 일종의 결함이므로 선택할 수 있는 뭔가가 아니기 때문에 오해의 소지 존재  
네트워크가 올바르게 동작하면 일관성과 완전한 가용성 모두 제공 가능  
따라서 CAP는 네트워크 분단이 생겼을 때 일관성과 가용성 중 하나를 선택하라는 의미  

<br>

공식적인 CAP 정리는 매우 범위가 좁아 하나의 일관성 모델(선형석)과 하나의 결함(네트워크 분단)만 고려  
네트워크 지연, 죽은 노드 등 다른 트레이드오프에 대한 것은 언급하지 않기 때문에 실용적인 가치는 거의 없음  

<br>

## 순서화 보장
순서화가 인과성을 보존하는데 도움을 줌  
인과성은 이벤트에 순서를 부과, 결과가 나타나기 전에 원인이 발생하고, 답변하기 전에 질문을 함  
시스템이 인과성에 의해 부과된 순서를 지키면 인과적으로 일관적(`causally coonsistent`)이라고 표현  

<br>

전체 순서(`total order`)는 어떤 두 요소를 비교할 수 있게 함  
그러나 수학적 집합은 항상 순서를 정할 수 있는 것은 아니기 때문에 부분적으로 순서가 정해짐  
- 선형성: 연산의 전체 순서를 정할 수 있음  
- 인과성: 두 연산 중 어떤 것도 다른 것보다 먼저 실행되지 않았다면 두 연산이 동시적, 전체 순서가 아닌 부분 순서 정의  

<br>

위의 정의에 따르면 선형성 데이터스토어에는 동시적 연산이 존재하지 않음  
하나의 타임라인이 있고 모든 연산은 그 타임라인을 따라서 전체 순서가 정해짐  
동시성은 타임라인이 잠시 갈라졌다가 다시 합쳐지는 것을 의미  

<br>

즉 선형성은 인과성을 내포하는 개념  
어떤 시스템이든지 선형적이라면 인과성도 올바르게 유지  
하지만 선형성이 인과성을 보존하는 유일한 방법이 아닌 다른 방법도 존재  
시스템을 선형적으로 만드는 성능 손해를 유발하지 않고도 인과적 일관성을 만족 가능  

<br>

## 인과적 의존성
비선형성 시스템이 인과적 일관성을 유지하기 위해서는 어떤 연산이 다른 연산보다 먼저 실행됐는지 알 필요 존재  
동시 실행된 연산은 어떤 순서로든 처리 가능하지만 모든 복제 서버가 그 순서에 맞게 처리되는 것을 보장  
인과적 순서 결정을 위해 데이터베이스는 애플리케이션이 데이터의 어떤 버전을 읽었는지 알 필요 존재  

<br>

### 일련번호 순서화
일련버호나 타임스탬프를 사용해서 이벤트의 순서 지정 가능  
이런 일련번호나 타임스탬프는 크기가 작고 전체 순서를 제공  
단일 리더 복제는 복제 로그가 인과성에 일관적인 쓰기 연산의 전체 순서를 정의  

<br>

### 비인과적 일련번호 생성기
단일 리더가 아닌 데이터베이스는 일련번호를 생성하는 방법이 명확해 보이지 않음  
- 각 노드가 자신만의 독립적인 일련번호 집합을 생성
- 각 연산에 타임스탬프를 추가해서 전체 순서 결정, 최종 쓰기 충돌 해소에서 사용
- 일련번호 블록을 미리 할당

<br>

위의 세가지 방법 모두 잘 동작하며, 카운터를 증가시키는 단일 리더보다 확장성이 좋음  
연산마다 근사 증가 일련번호를 생성하지만 생성한 일련번호가 인과성에 일관적이지 않음  
- 각 노드는 초당 연산수가 상이, 홀수용 노드와 짝수용 노드 카운터 간의 뒤처짐 발생 가능
- 물리적 시계에서 얻은 타임스탬프는 시계 스큐에 종속적이어서 인과성에 일관적이지 않을 가능성 존재
- 블록 할당자의 경우 한 연산이 높은 구간에서 일련번호를 부여받고 나중 연산이 낮은 구간에서 부여 받을 가능성 존재

<br>

### 램포트 타임스탬프

<img width="550" height="200" alt="lamport_timestamp" src="https://github.com/user-attachments/assets/73d37c61-09fe-49ed-8767-6b73486703ac" />

각 노드는 고유 식별자를 갖고 각 노드는 처리한 연산 개수를 카운터로 유지  
램포트 타임스탬프는 쉽게 `(카운터, 노드 ID)`의 쌍  
물리적 일 기준 시계와 아무 관련 없지만 전체 순서화를 제공  
두 타임스탬프 중 카운터가 큰 것이 타임스탬프가 크고, 카운터 값이 같으면 노드 ID가 큰 것이 큼  

<br>

핵심 아이디어는 모든 노드와 클라이언트가 지금까지 본 카운터 값 중 최댓값을 추적하고 모든 요청에 최댓값을 포함  
노드는 자신의 카운터 값보다 큰 카운터 값을 가진 요청이나 응답을 받으면 바로 자신의 카운터를 그 최댓값으로 증가  
동시 쓰기 감지에서 사용한 버전 벡터와 혼동되기 쉽지만 목적이 다름  
버전 벡터는 두 연산이 동시적인지 인과적으로 의존하는지 구별  
램포트 타임스탬프는 항상 전체 순서화를 강제  

<br>

### 타임스탬프 순서화 한계
램포트 타임스탬프는 연산의 전체 순서화가 사후에 성공하는 쪽을 결정하는데 효과적  
하지만 유일성 제약 조건 같은 생성 요청을 막 받고 그 요청이 성공해야 하는지 실패해야 하는지 당장 결정해야 할 때는 부족  
문제는 연산의 전체 순서가 모든 연산을 모은 후에야 드러난다는 것  

<br>

### 전체 순서 브로드캐스트
단일 CPU 코어에서만 프로그램이 실행된다면 연산의 전체 순서를 결정하기 쉬움  
이전의 타임스탬프나 일련번호를 사용한 순서화는 단일 리더 복제만큼 강력하지 않음  
문제는 처리량이 단일 리더가 처리할 수 있는 수준을 넘어설 때 어떻게 확장할 것인가  

<br>

파티션마다 단일 리더를 갖도록 파티셔닝을 한다면 각 파티션마다 순서를 유지  
하지만 여러 파티션에 걸친 일관성 보장을 제공할 수 없음  
모든 파티션에 걸친 전체 순서화도 가능하긴 하지만 이때도 추가적인 코디네이션이 필요  

<br>

전체 순서 브로드캐스트는 보통 노드 사이에 메시지를 교환하는 프로토콜로 기술  
- 신뢰성 있는 전달: 어떤 메시지도 손실되지 않음
- 전체 순서가 정해진 전달: 메시지는 모든 노드에 같은 순서로 전달

<br>

주키퍼나 etcd 같은 합의 서비스는 전체 순서 브로드캐스트를 실제로 구현  
이는 전체 순서 브로드캐스트와 합의 사이에 강한 연관이 존재  
모든 메시지가 쓰기를 나타내고 모든 복제 서버가 같은 쓰기 연산을 같은 순서로 처리  
해당 원리를 상태 기계 복제(`state machine replication`)이라고 표현  
전체 순서 브로드캐스트는 메시지가 전달되는 시점에 그 순서가 고정, 타임스탬프 순서화보다 강력  
또 다른 관점으로 로그를 만드는 방법 중 하나  

<br>

## 분산 트랜잭션과 합의
합의의 비공식적 목적은 단지 여러 노드들이 뭔가에 동의하게 만드는 것  
노드가 동의하는 것이 중요한 상황은 아래와 같음
- 리더 선출: 단일 리더 복젤ㄹ 사용하는 경우 모든 노드는 어떤 노드가 리더인지 동의가 필요
- 원자적 커밋: 여러 노드나 파티션에 걸친 트랜잭션은 원자성을 유지하기 위해 모든 노드가 트랜잭션 결과 동의 필요

<br>

합의 불가능성(`FLP`)은 어떤 노드가 죽을 위혐이 있다면 항상 합의에 이를 수 있는 알고리즘이 없다는 것을 증명  
어떤 시계나 타임아웃도 사용할 수 없는 결정적인 알고리즘을 가정하는 매우 제한된 모델  
즉 알고리즘이 타임아웃을 허용하거나 죽은 것으로 의심되는 노드를 식별 가능하다면 해결  
그러므로 합의 불가능성에 대한 결과는 이론적으로 중요하지만 분산 시스템은 보통 현실에서 합의를 달성 가능  

<br>

### 원자적 커밋과 2단계 커밋(2PC)
트랜잭션의 결과는 커밋 성공 또는 어보트  
원자성은 실패한 트랜잭션이 절반만 완료되거나 갱신된 상태로 데이터베이스를 어지럽히는 것을 방지  

<br>

단일 데이터베이스 노드에서 실행되는 트랜잭션에게 원자성은 흔히 저장소 엔진에서 구현  
그러나 파티셔닝된 데이터베이스에 다중 객체 트랜잰셕 같이 트랜잭션에 여러 노드가 관여한다면 원자성 보장이 필수  
- 어떤 노드들은 제약 조건 위반이나 충돌을 감지해서 어보트가 필요하게 하지만, 다른 노드는 성공적으로 커밋
- 어떤 커밋 요청은 네트워크 손실로 어보트되지만, 다른 커밋은 요청은 정상 전달
- 어떤 노드는 레코드 쓰기 중 다운으로 복구할 때 롤백되지만, 다른 노드는 성공적으로 커밋

<br>

### 2단계 커밋
여러 노드에 걸친 원자적 트랜잭션 커밋을 달성하는 알고리즘  
일부 데이터베이스는 2PC가 내부적으로 사용되고 XA 트랜잭션 또는 SOAP을 통해 사용가능  

<br>

단일 노드 트랜잭션에서는 존재하지 않는 코디네이터를 사용  
코디네이터는 종종 애플리케이션 프로세스 내에서 라이브러리 형태로 구현, 분리된 프로세스나 서비스도 가능  
애플리케이션이 커밋할 준비가 되면 코디네이터는 1단계를 시작  
- 모든 참여자가 커밋할 준비가 됐다는 응답을 보내면 코디네이터는 2단계에서 커밋 요청  
- 참여자 중 누구라도 준비가 안됐다는 응답을 하면 코디네이터는 2단계에서 모든 노드에 어보트 요청  

<br>

2단계 커밋은 코디네이터가 모든 참여자의 준비 완료 응답을 받고 커밋 또는 어보트를 결정하면 더이상 되돌릴 수 없음  
모든 참여자는 디스크 쓰기, 충돌, 제약 조건 위반 등 모든 것을 확인하는 것이 1단계  
코디네이터는 자신의 결정을 디스크에 트랜잭션 ID와 함께 로그로 기록, 이를 커밋 포인트  
만약 2단계 과정 중에 참여자가 문제가 발생하면 코디네이터는 그 참여자가 복구될 때까지 영원히 재시도  

<br>

### 코디네이터 장애
준비 요청 중 어떤게 실패하거나 타임아웃이 되면 코디네이터는 트랜잭션을 어보트  
커밋이나 어보트 요청이 실패하면 코디네이터는 무한히 재시도  
만약 코디네이터가 죽으면 참여자는 무한히 대기  
이 상태에 있는 참여자의 트랜잭션은 의심스럽다(`in doubt`) 또는 불확실하다(`uncertain`)고 표현  

<br>

<img width="550" height="200" alt="uncertain" src="https://github.com/user-attachments/assets/e984becc-7854-4a1d-a8f9-2dc0306056e4" />

코디네이터에게 듣지 않고 참여자는 커밋 또는 어보트를 결정할 수 없음  
이론상 참여자들끼리 통신해서 합의 가능하지만 이는 2PC 프로토콜의 일부가 아님  
결국 코디네이터가 복구되기를 대기, 이것이 커밋 포인트를 쓰는 이유  

<br>

### 3단계 커밋
2단계 커밋은 코디네이터가 복구하기를 대기하다가 멈출 수 있기 때문에 블로킹 원자적 커밋 프로토콜로 불림  
3PC는 지연에 제한이 있는 네트워크와 응답 시간에 제한이 있는 노드를 가정  
기약 없는 네트워크 지연과 프로세스 중단이 있는 대부분 실용적 시스템에서 3PC는 원자성을 보장하지 못함  
일반적으로 논블로킹 원자적 커밋은 완벽한 장애 감지기, 즉 신뢰성 잇는 메커니즘이 필요  
이런 까닭으로 코디네이터 장애와 관련된 알려진 문제가 존재함에도 2PC가 지속적으로 사용됨  

<br>

## 현실의 분산 트랜잭션
어떤 분산 트랜잭션은 무거운 성능 손해를 수반  
가장 많은 성능 비용은 장애 복구를 위해 필요한 부가적인 디스크 쓰기 강제와 네트워크 RTT  
분산 트랜잭션은 두 가지 종류가 흔히 사용
- 내부 분산 트랜잭션: 데이터베이스 노드 사이에 내부 트랜잭션을 사용(동일한 기술)
- 이종 분산 트랜잭션: 둘 혹은 그 이상의 다른 기술 사용(서로 다른 벤더)

<br>

### XA 트랜잭션
`X/Open XA(eXtended Architecture)`는 이종 기술에 걸친 2PC를 구현하는 표준  
여러 전통적인 관계형 데이터베이스와 메시지 브로커에서 지원  
네트워크 프로토콜이 아닌 트랜잭션 코디네이터와 연결되는 인터페이스를 제공하는 C API  
XA는 애플리케이션이 네트워크 드라이버나 클라이언트 라이브러리를 사용해 데이터베이스나 메시징 서비스와 통신한다고 가정  
애플리케이션 프로세스가 죽으면 코디네이터도 함께 사라짐  
코디네이터의 로그는 애플리케이션 서버 로컬 디스크에 존재하며 해당 서버가 재시작돼야 복구 가능  

<br>

### 의심스러운 상태에서의 잠금 유지 문제
데이터베이스 트랜잭션은 보통 더티 쓰기를 막기 위해 그들이 변경한 로우에 로우 수준 잠금을 획득  
데이터베이스는 트랜잭션의 잠금을 해제할 수 없기 때문에 의심스러운 상태에 있는 동안 잠금을 유지  
이런 잠금 유지는 다른 트랜잭션이 해당 로우를 변경 불가 또는 접근 금지를 강제  

<br>

### 코디네이터 장애 복구
이론상 코디네이터가 죽은 후 재시작하면 로그로부터 그 상태를 깨끗하게 복구하고 의심스러운 트랜잭션 해소 가능  
하지만 현실에서는 고아가 된(`orphaned`) 의심스러운 트랜잭션, 즉 코디네이터가 결과를 결정할 수 없는 트랜잭션이 발생  
이런 트랜잭션은 자동으로 해소 불가해서 잠금을 유지하고 다른 트랜잭션을 지속적으로 차단  

<br>

데이터베이스 서버를 재부팅해도 문제 해결이 불가능, 2PC의 올바른 구현은 재시작을 하더라도 의심스러운 트랜잭션 잠금을 유지  
해당 문제는 관리자가 직접 수동으로 트랜잭션을 커밋하거나 롤백, 모든 참여자에 적용  
여러 XA 구현에는 의심스러운 트랜잭션을 어보트하거나 커밋할지를 일방적으로 결정할 수 있도록 경험적 결정(`heuristic decision`) 존재  
경험적 결정은 2단계 커밋 약속 체계를 위반하기 때문에 엄밀히 따지면 원자성을 깰 가능성 존재  
따라서 경험적 결정은 평상시가 아닌 큰 장애 상황을 벗어나고자 할 때만 사용되도록 의도  

<br>

### 분산 트랜잭션의 제약
XA 트랜잭션의 핵심 구현은 트랜잭션 코디네이터 자체가 일종의 데이터베이스여야 한다는 점  
- 코디네이터가 복제되지 않고 단일 장비에서만 실행되는 경우 SPOF
- 코디네이터의 로그로 인해 애플리케이션의 무상태가 깨짐
- XA는 광범위한 데이터 시스템과 호환되기 때문에 최소 공통 분모가 될 필요성 존재  

<br>

## 내결함성을 지닌 합의
하나 이상의 노드들이 값을 제안할 수 있고 합의 알고리즘이 그 값 중 하나를 경정  
합의 알고리즘은 아래 송성을 만족
- 균일한 동의: 어떤 두 노드도 다르게 결정하지 않음
- 무결성: 어떤 노드도 두 번 결정하지 않음
- 유효성: 한 노드가 어떤 값을 결정한다면 그 값은 어떤 노드에서 제안될 것
- 종료: 죽지 않은 모든 노드는 결국 어떤 값을 결정

<br>

합의 알고리즘은 모두 같은 결과로 결정하며 한 번 결정되면 번복할 수 없음  
내결함성이 상관없다면 처음 세 개의 속성을 만족시키는 것은 쉬움  
종료 속성은 합의 알고리즘이 아무것도 하지 않는 상태를 방지하며 진행해야 한다고 규정  
즉 어떤 노드들에 장애가 발생하더라도 다른 노드들은 여전히 결정을 내려야 함  
합의 시스템 모델은 노드가 죽으면 그 노드는 결코 돌아오지 않는다고 가정  

<br>

종료 속성은 죽거나 연결할 수 없는 노드 대수가 절반 미만이라는 가정에 종속  
대규모 중단이 발생하면 시스템이 요청 처리를 할 수 없지만 유효하지 않은 결정을 내려서 합의 시스템을 오염시키지 않음  
대부분의 합의 알고리즘은 비잔틴 결함이 없다고 가정  
즉 노드가 프로토콜을 올바르게 따르지 않으면 프로토콜의 안전성 속성이 깨질 가능성 존재   

<br>
