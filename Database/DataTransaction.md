# 트랜잭션
트랜잭션은 애플리케이션에서 몇 개의 읽기와 쓰기를 하나의 논리적 단위로 묶는 방법  
트랜잭션은 전체가 성공하거나 실패  

## ACID
트랜잭션이 제공하는 안전성 보장은 흔히 원자성, 일관성, 격리성, 지속성을 의미  

<br>

### 원자성(Atomicity)
일반적으로 더 작은 부분으로 쪼갤 수 없는 것을 의미  
여러 쓰기 작업이 하나의 원자적인 트랜잭션으로 묶여있고 결함 때문에 일부가 완료될 수 없다면 전체가 어보트  
즉 오류가 발생했을대 트랜잭션을 어보트하고 해당 트랜잭션에서 기록한 모든 내용을 취소하는 능력  

<br>

### 일관성(Consistency)
일관성의 아이디어는 애플리케이션 불변식 개념에 의존  
데이터베이스에서 불변식을 위반하는 잘못된 데이터를 쓰지 못하도록 방지 불가능  
데이터베이스에서 방지 가능한 부분은 그저 외래키 제약 조건이나 유일성 제약 조건  
즉 일관성은 정확하게는 애플리케이션의 속성  

<br>

### 격리성(Isolation)
동시에 실행되는 트랜잭션은 서로 격리된다는 것을 의미  
고전적인 곳에서는 격리성을 직렬성이라는 용어로 공식화  
각 트랜잭션이 전체 데이터베이스에서 실행되는 유일한 트랜잭션인 것처럼 동작한다는 것을 의미  
하지만 직렬성 격리는 성능 손해를 동반하기 때문에 현실에서는 거의 사용되지 않음  
실제로는 직렬성보다 보장이 약한 스냅샷 격리를 구현  

<br>

### 지속성(Durability)
트랜잭션이 성공적으로 커밋됐다면 하드웨어 결함이 발생하거나 데이터베이스가 죽더라도 기록한 모든 데이터는 손실되지 않는다는 보장  
비휘발성 저장소에 기록됐다는 뜻으로 일반적으로 데이터 구조가 오염됐을 때 복구 가능하게 해주는 쓰기 전 로그나 비슷한 수단을 동반  

<br>

## 완화된 격리 수준
동시성 문제는 트랜잭션이 다른 트랜잭션에서 동시에 변경한 데이터를 읽거나 두 트랜잭션이 동시에 같은 데이터를 변경할때만 발생  
데이터베이스는 오랫동안 트랜잭션 격리를 제공함으로써 동시성 문제를 감추려 시도  

<br>

### 커밋 후 읽기
가장 기본적인 수준의 트랜잭션 격리는 커밋 후 읽기  
아직 커밋되거나 어보트되지 않은 다른 트랜잭션의 데이터를 보는 더티 읽기를 방지  

<br>

<img width="550" height="150" alt="dirty_read" src="https://github.com/user-attachments/assets/f6380df1-6606-4b50-abde-53ab9202bbfd" />

가장 흔한 구현 방법은 데이터베이스의 로우 수준 잠금을 사용  
트랜잭션에서 특정 객체를 변경하고 싶다면 먼저 해당 객체에 대한 잠금을 획득  
트랜잭션이 커밋되거나 어보트될 때까지 잠금을 보유하고 다른 트랜잭션은 잠금을 획득할 수 없음  

<br>

### 스냅샷 격리와 반복 읽기

<img width="550" height="200" alt="nonrepetable_read" src="https://github.com/user-attachments/assets/d05b9446-c69f-438a-9c0b-48ceddc037bd" />

트랜잭션의 시작과 끝 시점에 질의의 결과가 다른 비반복 읽기(`nonrepeatable read`)나 읽기 스큐(`read skew`) 문제 발생 가능  
스냅샷 격리를 통해 각 트랜잭션은 데이터베이스의 일관된 스냅샷으로부터 조회 가능  
스냅샷 격리는 백업이나 분석처럼 실행하는데 오래 걸리며 읽기만 실행하는 질의에 유용  

<br>

쓰기를 실행하는 트랜잭션은 같은 객체에 쓰는 다른 트랜잭션의 진행을 차단, 하지만 읽기는 아무 잠금도 필요 없음  
핵심 원리는 읽는 쪽에서 쓰는 쪽을, 쓰는 쪽에서 읽는 쪽을 결코 차단하지 않음  
데이터베이스가 객체의 여러 버전을 함께 유지하는 다중 버전 동시성 제어(`MVCC`)  

<br>

### 갱신 손실 방지
커밋 후 읽기와 스냅샷 격리 수준은 주로 동시에 실행되는 쓰기 작업을 읽기 전용 트랜잭션이 어떻게 보는지 보장  
하지만 동시에 실행되는 쓰기 트랜잭션 사이에 발행사는 충돌은 갱신 손실(`lost update`) 문제  
갱신 손실 문제는 애플리케이션이 데이터베이스에서 값을 읽고 변경한 후 변경된 값을 다시 쓸때(`read-modify-write`) 발생 가능  
여러 데이터베이스에서 원자적 갱신 연산을 제공  

<br>

## 직렬성
직렬성 격리는 보통 가장 강력한 격리 수준으로 여겨짐  
여러 트랜잭션이 병렬로 실행되더라도 최종 결과는 동시성 없는 직렬로 실행될 떄와 같도록 보장  

<br>

### 실제적인 직렬 실행
동시성 문제를 피하는 가장 간단한 방법은 동시성을 완전히 제거하는 것  
트랜잭션을 순차적으로 실행하는 방법은 레디스, 데이토믹에서 구현  
잠금을 코디네이션하는 오버헤드를 피할 수 있기 때문에 동시성을 지원하는 시스템보다 성능이 좋은 경우 존재  

<br>
