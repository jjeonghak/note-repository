# 복제
네트워크로 연결된 여러 장비에 동일한 데이터의 복사본을 유지  
지리적으로 사용자와 가깝게 데이터를 유지해 지연 시간 감소 가능  
시스템 일부에 장애 발생시 지속적으로 동작할 수 있게 해서 가용성 높임  
읽기 질의를 제공하는 장비의 수를 확장해 읽기 처리량 높임  

<br>

## 리더와 팔로워

<img width="550" height="200" alt="leader_based_replication" src="https://github.com/user-attachments/assets/c546e1cd-8116-41ad-bb81-d16ae99affd6" />

데이터의 동기화를 위해서 가장 일반적인 리더 기반 복제(`leader-based replication`) 사용  
쓰기 요청은 먼저 리더에게 보낸 후 데이터 변경을 복제 로그나 변경 스트림의 일부로 팔로워에게 전송  

<br>

### 동기식 대 비동기식 복제
동기식 복제의 장점은 팔로워가 리더와 일관성 있게 최신 데이터 복사본 보유 보장  
하지만 동기 팔로워가 응답하지 않는 경우 리더는 블락되어 대기  
현실적으로 동기식 복제를 사용하려면 보통 팔로워 하나는 동기식, 그 밖에는 비동기식으로 반동기식(`semi-synchronous`) 설정을 사용  
보통 리더 기반 복제는 완전히 비동기식으로 구성  

<br>

### 새로운 팔로워 서정
새로운 팔로워를 설정해야 하는 경우 리더의 데이터 복제본을 정확히 보유한 것인지 보장 어려움  
데이터베이스를 잠가서 디스크 파일을 일관성 있게 복사 가능하지만 가용성 목표에 부합하지 않기 때문에 아래 과정 수행
1. 리더의 데이터베이스 스냅샷을 일정 시점에 불러옴
2. 스냅샷을 새로운 팔로워 노드에 복사
3. 팔로워는 리더에 연결해 스냅샷 이후 발생한 데이터 변경을 요청
4. 스냅샷 이후 데이터 변경 미처리분(`backing`)을 모두 처리 완료하면 모두 따라잡음

<br>

### 노드 중단 처리
각 팔로워는 리더로부터 수신한 데이터 변경 로그를 로컬 디스크에 보관  
보관된 로그에서 결함이 발생하기 전 처리한 마지막 트랜잭션을 탐색  
팔로워는 리더에 연결해 연결이 끊어진 동안 발생한 데이터 변경을 모두 요청 가능  

<br>

리더의 장애는 팔로워 중 하나를 새로운 리더로 승격해서 해결 가능하지만 아래 문제 발생  
비동기식 복제를 사용한다면 새로운 리더는 이전 리더가 실패하기 전 일부 쓰기를 수신하지 못할 가능성 존재  
이전 리더의 복제되지 않은 쓰기를 단순히 폐기하는 방법이 일반적이지만 클라이언트의 기대를 저버리게 됨  
스플릿 브레인 현상이 발생해서 쓰기 충돌 가능성 존재  
리더가 죽었다고 판단 가능한 적절한 타임아웃 설정이 어려움  

<br>

### 복제 로그 구현
`구문 기반 복제`는 리더가 모든 쓰기 요청을 기록하고 실행한 후 팔로워에게 구문 로그를 전송  
MySQL의 경우 구문에 비결정성이 있다면 로우 기반 복제(`row-based replication`)로 변경해서 사용  
- 현재 날짜와 시간을 얻기 위한 `NOW()`나 임의 숫자를 얻기 위한 `RAND()` 같은 비결정적 함수를 호출하는 경우
- 자동증가 칼럼을 사용하는 구문이나 데이터 의존 구문은 각 복제 서버에서 정확히 같은 순서로 실행 보장 어려움
- 부수 효과를 가진 트리거, 스토어드 프로시저, 사용자 정의 함수는 완벽하게 결정적이지 않으면 또 다른 부수 효과 발생

<br>

모든 쓰기는 로그에 기록되기 때문에 `쓰기 전 로그 전송`으로 완전히 동일한 로그를 사용해 복제 서버 구축 가능  
리더는 디스크에 로그를 기록하고 팔로워에게 로그를 전송  
postgreSQL과 오라클 등에서 사용  
로그가 저수준의 데이터를 기술하기 때문에 저장소 엔진과 밀접하게 의존, 데이터베이스 저장소 형식을 다른 버전으로 변경 불가  

<br>

`로우 기반 로그 복제`는 복제 로그를 저장소 엔진 내부와 분리하기 위한 대안으로 다른 로그 형식을 사용  
저장소 엔진의 물리적 데이터 표현과 구별하기 위해 논리적 로그를 사용  
논리적 로그 형식은 외부 애플리케이션이 파싱하기 쉽기 때문에 해당 기술을 변경 데이터 캡처(`change data capture`)라고 부름  

<br>

## 복제 지연 문제
읽기 확장 아키텍처에서는 간단히 팔로워를 추가함으로써 읽기 전용 요청 처리를 위한 용량 증가 가능  
해당 방식은 비동기식 복제에서만 동작하기 때문에 뒤처진 팔로워의 경우 이전 버전의 데이터 조회 가능성 존재  
리더와 팔로워에 동일한 질의를 수행해도 서로 다른 결과를 얻는 일시적 불일치 발생 가능  
결국 팔로워는 리더와 일치하게 되기 때문에 최종적 일관성은 유지  

<br>

<img width="550" height="250" alt="read_after_write" src="https://github.com/user-attachments/assets/1028b6b8-9396-48b7-9fb2-36583d125100" />

항상 자신이 제출한 쓰기에 대해 최신 결과를 볼 수 있음을 보장하는 것이 쓰기 후 읽기 일관성(자신의 쓰기 읽기 일관성)   
- 사용자가 수정한 내용을 읽을 때는 리더에서만 조회, 그 밖에서는 팔로워에서 조회  
- 클라이언트가 가장 최근 쓰기 타임스탬프를 기억하고, 최소한 해당 타임스탬프까지 갱신 반영한 복제 서버가 있을 때까지 대기  

<br>

동일한 사용자가 여러 디바이스로 서비스 접근할 경우 디바이스 간 쓰기 후 읽기 일관성이 필요  
- 사용자의 마지막 갱신 타임스탬프 기억 접근 방식이 어려움
- 복제 서버가 여러 데이터센터 간에 분산된 상태면 동일한 데이터센터로 라우팅된다는 보장이 없음  

<br>

### 단조 읽기
사용자가 각기 다른 복제 서버에서 여러 읽기를 수행할 때 결과가 `과거로 돌아가는 문제` 발생 가능  
단조 읽기(`monotonic read`)는 이런 종류의 이상 현상이 발생하지 않음을 보장  
강한 일관성보다는 덜한 보장이지만 최총적 일관성보다는 더 강한 보장  

<br>

<img width="550" height="300" alt="monotonic_read" src="https://github.com/user-attachments/assets/97c82e32-3250-4e67-aa7d-444637ced867" />

각 사용자의 읽기가 항상 동일한 복제 서버에서 수행하게끔 강제  
예를 들어 복제본을 임의로 선택하지 않고 사용자 ID의 해시 기반으로 복제 서버를 선택 가능  

<br>

### 일관된 순서로 읽기
인과성 위반 현상을 막기 위해서는 일관된 순서 읽기 필요  
파티셔닝된(샤딩된) 데이터베이스에서 발생하는 특징적인 문제  

<br>

<img width="550" height="300" alt="consistent_prefix_read" src="https://github.com/user-attachments/assets/849a1cfd-ab92-4b5c-9400-f4d392f58125" />

일련의 쓰기가 특정 순서대로 발생한다면 이 쓰기를 읽는 모든 사용자는 같은 순서로 쓰여진 내용 죄회 보장  
분산 데이터베이스에서 서로 다른 파티션은 독립적으로 동작하므로 쓰기의 전역 순서는 존재하지 않음  
서로 인과성이 있는 쓰기가 동일한 파티션에 기록되게끔 하는 방법으로 해결가능하지만 효율적이지 않음  

<br>

## 다중 리더 복제
리더 기반 복제는 리더가 하나만 존재하고 모든 쓰기가 해당 리더를 거쳐야하기 때문에 리더 장애에 치명적  
다중 리더의 경우 각 리더는 동시에 다른 리더의 팔로워 역할도 수행  
다중 데이터센터를 운영하는 경우 각 데이터센터마다 리더가 존재 가능  

<br>

<img width="550" height="250" alt="multi_leader_replication" src="https://github.com/user-attachments/assets/12314a5f-be97-4557-978f-4c5987e932f3" />

모든 쓰기는 로컬 데이터센터에서 처리한 후 비동기 방식으로 다른 데이터센터에 복제  
각 데이터센터는 다른 데이터센터와 독립적으로 동작하고 장애 발생한 데이터센터가 회복한 후 복제를 따라잡음  
동일한 데이터를 다른 두 개의 데이터센터에서 동시에 변경 가능, 이때 발생하는 쓰기 충돌을 반드시 해소해야 함  

<br>

### 쓰기 충돌

<img width="550" height="250" alt="write_conflict" src="https://github.com/user-attachments/assets/826ea6c3-7d59-4503-b1ea-e77e8f35cf5e" />

단일 리더의 경우 첫번째 쓰기가 완료될 때까지 두번째 쓰기를 차단해 대기하게 하거나 재시도하도록 유도  
다중 리더의 경우 두 쓰기 모두 성공하며 충돌은 이후 특정 시점에서 비동기로만 감지  

<br>

충돌을 처리하는 가장 간단한 전략은 충돌을 방지하는 것  
특정 레코드의 모든 쓰기를 동일한 리더를 거치도록 애플리케이션이 보장한다면 충돌을 방지 가능  
특정 사용자의 요청을 동일한 데이터센터로 항상 라우팅한다면 사용자 관점에서는 단일 리더  
하지만 때때로 데이터센터 장애나 지역 이동에 의한 리더 변경이 필요한 경우 충돌 회피가 실패  

<br>

### 일관된 상태 수렴
다중 리더의 경우 쓰기 순서가 정해지지 않아 최종값이 무엇인지 명확하지 않음  
단순하게 각 복제 서버가 쓰기를 본 순서대로 적용한다면 데이터베이스는 일관성 없는 상태 유지  
따라서 데이터베이스는 수렴(`convergent`) 방식으로 충돌을 해소  
- 각 쓰기에 고유 ID를 부여하고 가장 높은 ID를 가진 쓰기를 채택
- 타임스탬프를 사용하는 경우 최종 쓰기 승리(`last write wins`)라고 표현
- 각 복제 서버에 고유 ID를 부여하고 높은 숫자의 복제 서버에서 생긴 쓰기가 우선적으로 적용
- 명시적 데이터 구조에 충돌을 기록해서 모든 정보를 보존(이후 사용자 또는 애플리케이션이 충돌을 직접 해소)

<br>

### 다중 리더 복제 토폴로지

<img width="550" height="150" alt="replication_topology" src="https://github.com/user-attachments/assets/936bfe57-e39f-4a37-924a-11705625ca4b" />

복제 토폴로지는 쓰기를 한 노드에서 다른 노드로 전달하는 통신 경로  
가장 일반적인 토폴로지는 전체 연결(`all-to-all`), 모든 리더가 각자의 쓰기를 다른 모든 리더에게 전송  
원형 토폴로지(`circular topology`)는 각 노드가 하나의 노드로부터 쓰기를 받고 이 쓰기를 다른 한 노드에 전달  
별 모양 토폴로지는 지정된 루트 노드 하나가 다른 모든 노드에 쓰기를 전달  

<br>

<img width="550" height="250" alt="all_to_all_network_issue" src="https://github.com/user-attachments/assets/993e07f5-333d-487f-9232-1969c4b99d7d" />

원형과 별 모양 토폴로지는 하나의 노드에 장애가 발생하면 다른 노드 간 복제 메시지 흐름에 영향  
반면 전체 열결 토폴로지는 일부 네트워크 연결이 다른 연결보다 빠르자면 일부 복제 메시지 추월 발생 가능  
일관된 순서로 읽기에서의 인과성 문제로 추월 발생시 삽입과 갱신의 순서 역전 발생 가능  
이런 이벤트를 올바르게 정렬하기 위해 버전 벡터 기법을 사용 가능  

<br>

## 리더 없는 복제
리더의 개념을 버리고 모든 복제 서버가 클라이언트로부터 쓰기를 직접 받을 수 있도록 허용  
아마존의 다이나모 시스템에서 사용한 후 데이터베이스용 아키텍처로 유행  

<br>

### 노드 다운시 쓰기

<img width="550" height="250" alt="replication_without_leader" src="https://github.com/user-attachments/assets/9c3943f4-9a6b-4159-b632-cea84d4251e2" />

리더 없는 설정의 경우 장애 복구가 필요하지 않음  
클라이언트는 복제 서버 중 하나가 쓰기를 놓친 사실을 무시  
회복된 노드가 다시 온라인 상태가 된 경우 오래된 값으로 응답  
클라이언트는 데이터베이스를 읽을 때 읽기 요청을 병렬로 여러 노드에 전송  
이때 여러 노드에서의 결과를 버전 숫자를 이용해 어떤 값이 최신 값인지 결정  

<br>

### 읽기 복구와 안티 엔트로피
사용 불가능한 노드가 온라인 상태가 된 후 누락된 쓰기를 따라잡는 방법 필요  
- 읽기 복구: 클라이언트가 오래된 값을 감지하고 이를 복제 서버에 새로운 값을 다시 기록
- 안티 엔트로피 처리: 추가적인 백그라운드 프로세스를 두고 복제 서버 간 데이터 차이를 지속적으로 탐색

<br>

### 읽기와 쓰기를 위한 정족수
`n`개의 복제 서버가 존재할 때 모든 쓰기는 `w`개의 노드에서 성공해야 쓰기가 확정되고 모든 읽기는 최소한 `r`개의 노드에 질의  
`w + r > n`이면 읽을 때 최신 값을 얻을 것으로 기대, 이는 쓰기 노드 셋과 읽기 노드 셋이 겹치기 때문  
이런 `r`과 `w`를 따르는 읽기와 쓰기를 정족수 읽기와 쓰기라고 표현  
일반적으로 `n`은 홀수로 설정하고 `w = r = (n + 1) / 2`로 설정
